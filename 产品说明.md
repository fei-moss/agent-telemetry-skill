# Agent Telemetry — 产品说明与安装指导

面向**接入本方案的产品方**:一页讲清这个组件能做什么、数据长什么样、以及如何
让你的用户把它装上并把数据上报到你的后端。

> 配套文档:终端用户启用步骤见 [用户侧使用文档.md](用户侧使用文档.md);后端搭建/
> token/去重见 [使用说明.md](使用说明.md);接入协议见 [docs/PROTOCOL.md](docs/PROTOCOL.md);
> 架构与失败语义见 [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)。

---

## 1. 这是什么

一个**本地优先的 Agent 观测组件**。用户在本地跑的 Agent(Claude Code、Codex CLI、
OpenClaw、Hermes 等)在执行任务时产生的**思考、进度、工具调用、模型调用、决策、
错误**,会被采集并上报到你提供的统一后端,供你做实时展示、排障、分析与计费。

**核心理念**:不依赖模型"自觉上报",而是挂到各 runtime 的确定性出口(hook /
plugin / 会话日志),保证全量、不可绕过的采集。

---

## 2. 核心能力

| 能力 | 说明 |
| --- | --- |
| **四框架适配** | Claude Code、Codex、OpenClaw、Hermes,各用其 runtime 的确定性出口采集 |
| **富内容采集** | 模型的**思考(reasoning)、进度(message)、工具调用与结果全文** —— 可直接渲染成给人看的对话时间线 |
| **近实时上报** | 日志增量监听 + 短周期上报,新动作几秒内到后端(可做前端实时展示) |
| **统一 trace** | 同一会话的所有事件(思考/工具/决策)汇入同一条 trace,跨进程串联 |
| **隐私脱敏(三档)** | 默认清洗密钥保留正文 / 完全原文(`disable_redaction`)/ 可调截断 |
| **断网不丢** | 本地 spool 持久化,断网暂存、恢复补传,进程崩溃后自动回收 |
| **不影响宿主** | 所有采集入口异常隔离,绝不阻塞、拖慢或拖垮用户的 Agent |
| **标准 OTLP** | 后端用标准 OpenTelemetry OTLP/HTTP,可接 Collector / Langfuse / Phoenix / ClickHouse |
| **生产托管** | systemd 服务化,开机自启、崩溃重启 |

### 采集分层与可信度

每条数据带 `telemetry.collection_layer` 标签,标识来源与可信度:

| 层 | 来源 | 内容 | 可信度 |
| --- | --- | --- | --- |
| `hook` / `plugin` | runtime hook / 插件 | 工具调用、模型调用、会话生命周期 | 高(确定性,不可绕过) |
| `log_watch` | 会话日志监听 | **思考 + 进度 + 工具 IO 全文** | 高(读完整 transcript) |
| `model_reported` | 模型经 CLI 自报 | 决策理由、置信度 | 补充(模型自述) |

> 给人看的"思考/进度/工具调用"主要来自 **`log_watch` 层**。

## 3. 数据长什么样

上报到后端的是标准 OTLP span(对齐 OTel GenAI 语义)。开启富内容后,后端可渲染成:

```
[思考] 用户问 BTC 实盘是否调仓。需要先查当前持仓和最新价格，再根据波动判断。
[进度] 我先查一下你当前的 BTC 持仓和最新价格。
[工具] (terminal) {"output": "持仓: 0, 账户余额: 9540 USDT, BTC=68250"}
[思考] 当前空仓，BTC 在区间震荡、无突破信号。稳健策略下不应追单。
[进度] 本轮结论：观望，不调仓。
```

每个 span 携带:`service.name`、`tenant.id`、`session.id`、`gen_ai.*`(模型/工具/
token 用量)、`telemetry.collection_layer`、时间戳。完整字段见
[docs/PROTOCOL.md](docs/PROTOCOL.md)。

## 4. 如何让你的用户装上(三步)

### 第 1 步:把安装包发给用户

用打包脚本生成分发包(自动排除测试/缓存,附带 `INSTALL.md`):

```bash
python3 scripts/build_package.py        # 产出 dist/agent-telemetry-skill-<版本>.tar.gz / .zip
```

把 `tar.gz` 或 `zip` 连同 sha256 一起发给用户。

### 第 2 步:用户一键安装

用户解压后:

```bash
tar xzf agent-telemetry-skill-<版本>.tar.gz
cd agent-telemetry-skill-<版本>
python3 scripts/setup.py --auto          # 自动探测已装 runtime 并安装对应适配器
```

`--auto` 会检测用户装了 Claude Code / Codex / OpenClaw / Hermes 中的哪些,自动
安装对应的 hook / 插件 / 日志监听器,并初始化 `~/.agent-telemetry/config.json`。

### 第 3 步:配置上报到你的后端

你给每个用户分配四个值并让其写入(env 或 config.json,env 优先):

```bash
python3 scripts/setup.py \
  --endpoint "https://telemetry.你的域名/v1/traces" \
  --token    "<为该用户签发的接入 token>" \
  --tenant   "<租户ID>" \
  --service  "<服务名>"
```

验证:

```bash
python3 scripts/setup.py --status        # 查看安装与配置状态
PYTHONPATH=. python3 -m agent_telemetry_skill.cli demo   # 发测试 trace,密钥应显示 [REDACTED]
```

## 5. 启用"富内容 + 近实时展示"

若你的产品要在前端实时展示用户 Agent 的思考/进度/工具调用,在用户的
`~/.agent-telemetry/config.json` 开启:

```json
{
  "endpoint": "https://telemetry.你的域名/v1/traces",
  "token": "<接入 token>",
  "tenant": "<租户ID>",
  "service": "<服务名>",
  "capture_narrative": true,
  "capture_content": true,
  "max_content_chars": 4000
}
```

然后以**近实时**模式运行日志监听器(几秒一轮):

```bash
PYTHONPATH=. python3 scripts/watch_sessions.py --runtime hermes --interval 5
# 或 --runtime claude-code / codex / all
```

生产环境建议把它做成 systemd 服务(见第 7 节),开机自启、崩溃重启。

## 6. 隐私与安全(给最终用户的承诺)

- **默认不上传**完整 prompt、模型回复、工具输出、文件内容里的**密钥**;
  `api_key`/`token`/`password`、`sk-*`/AWS/GitHub/Slack key、JWT、PEM 私钥块,
  以及命令行内联凭证(`sshpass -p`、`--password`、`curl -u`、URL `user:pass@`)
  都会被替换为 `[REDACTED]`。
- **三档可调**(用户/产品方自选):
  - 默认:留正文、清洗密钥(适合给人看,又不泄露凭证)
  - `"capture_content": false`:连正文也省略,只留结构与字数(最小化)
  - `"disable_redaction": true`:**完全原文**直通(密钥也保留,谨慎使用)
- **`AGENT_TELEMETRY_ENABLED=0`** 一键全局静默。
- 数据传输用 token 鉴权;**生产务必上 TLS**(反代到 ingest 或 ingest 走 443)。
- `~/.agent-telemetry/config.json` 建议 `chmod 600`。

## 7. 生产部署(产品方后端 + 用户侧服务)

**后端 ingest**(你这边,带 token 鉴权 + 按天落盘,systemd 托管):

```bash
# server/ingest_server.py:强制 Bearer 鉴权、body 上限、健康检查、不存明文 token
INGEST_TOKENS=<token[,token2]> python3 server/ingest_server.py --host 0.0.0.0 --port 4318
```

生产建议接 OpenTelemetry Collector(retry / 持久队列 / 二次脱敏 / tail sampling),
并按 token 在服务端裁定真实 tenant(不信任客户端上报的 `tenant.id`)。详见
[使用说明.md](使用说明.md)。

**用户侧采集服务**(systemd,开机自启):

```ini
# /etc/systemd/system/agent-telemetry-watcher.service
[Service]
Environment=PYTHONPATH=/opt/agent-telemetry-skill
Environment=HOME=/root
ExecStart=/usr/bin/python3 /opt/agent-telemetry-skill/scripts/watch_sessions.py --runtime hermes --interval 5
Restart=always
```

```bash
systemctl enable --now agent-telemetry-watcher.service
```

## 8. 常见问题

**会拖慢用户的 Agent 吗?**
不会。所有采集入口异常隔离,失败只暂存本地、绝不阻塞宿主;网络问题也不影响。

**断网会丢数据吗?**
不会。先持久化到本地 spool,恢复后自动补传。

**同一动作会重复上报吗?**
同一 runtime 只应启用一层(hook 或 watcher)。若两层都开,`execute_tool` 会重复,
后端应按 `span_id` 去重、按 `collection_layer` 取优(`hook`/`plugin` > `log_watch`
> `model_reported`)。

**思考(reasoning)一定能拿到吗?**
取决于模型是否把思考写进日志。MiniMax、开了可见 extended thinking 的模型有;部分
模型只留签名(空)。进度(message)与工具 IO 稳定可得。

**支持哪些后端?**
任何标准 OTLP/HTTP 端点:OpenTelemetry Collector、Langfuse、Phoenix、Tempo、
ClickHouse,或自建 ingest。协议见 [docs/PROTOCOL.md](docs/PROTOCOL.md)。
